# «Живі» події

> Я тут трохи забіжу наперед, тож якщо щось стало незрозуміло, відкладіть цей розділ «на потім».

Варто звернути увагу на ще одну задачку, яка дуже часто ставиться перед розробником – це додавання обробників подій для елементів, які додаються на сторінку динамічно. Мабуть, треба навести приклад подібної задачі:

_– У нас є HTML-сторінка, на якій всі внутрішні посилання будуть підвантажуватися через AJAX, це твердження справедливе і для підвантажуваного HTML теж._

Перша умова вирішується просто:

```javascript
$("a[href^=\\/]").on("click", function() {
    var url = $(this).attr("href");
    $("body").load(url + " body > *");
    return false;
});
```

> Для наочності домовимося, що внутрішні посилання містять відносні шляхи від кореня сайту.

З другою умовою трішки складніша ситуація, але теж цілком вирішувана:

```javascript
$("body").on("click", "a[href^=\\/]", function() {
    var url = $(this).attr("href");
    $("body").load(url + " body > *");
    return false;
});
```

Відмінностей не так вже й багато, проясню те, що відбувається:

* першим ділом на елемент `<body>` буде повішено обробник події `click`
* цей обробник буде спрацьовувати тільки в тому випадку, коли подія буде стосуватися елемента&#x20;

Робота цієї схеми базується на «спливанні» подій, тож, використовуючи метод `event.stopPropagation()`, ви зможете запобігти виконанню «живих» обробників.

> Ліричний відступ до історії: жив-був колись плагін для jQuery, називався «live», дозволяв він вішати обробники на елементи DOM-дерева, яких ще немає (підвантажувані через AJAX або ще як), а потім ~~він помер~~ його внесли до самого ядра. Метод `live()` на той час працював лише з `document`. Потім з'явився метод `delegate()`, який навчився вішати обробник на довільний елемент, а потім і він був поглинутий методом `on()`. Тож не лякайтеся сильно, якщо зустрінете старий метод `live()`, адаптувати під нові версії jQuery його буде не так вже й складно (ну, я на це сподіваюся).
